## 3. Build & Deploy Commands

All `npm` commands run from `site/` — there is no root-level `package.json`.

```bash
# Local development
cd site && npm run dev           # dev server at localhost:4321 (hot reload)
cd site && npm run build         # production build + Pagefind search index
cd site && npm run preview       # preview the production build locally

# Content pipeline (run from repo root)
python3 site/scripts/prepare-content.py   # Manual → site/src/content/recipes/ (no build)

# Full deploy pipeline (from repo root)
bash site/scripts/deploy.sh               # content prep + build + rsync to FastComet
```

**Critical:** `npm run build` runs `astro build` followed by `npx pagefind`. Both must succeed for site search to work. Do not run `astro build` directly when building for deployment — use `npm run build`.

**Critical:** Pushing to GitHub does **not** deploy the site. Only the rsync in `deploy.sh` updates the live site. Never report the site as deployed until rsync completes successfully.

## 4. Content Pipeline

`site/scripts/prepare-content.py` transforms `The Manual/` source files into the Astro content collection at `site/src/content/recipes/`.

### What It Does

1. **Strips the leading H1** — every folio starts with `# Café Athena - Title`. The script removes this because `RecipeLayout.astro` renders the title from frontmatter; leaving it doubles the title on the page.

2. **Extracts `## Keywords` and `## Category` from the body** — pulls these sections out, maps them into YAML frontmatter fields (`keywords[]`, `cuisine`, `style`, `family`, `course`, `dietary`), then strips the sections from the rendered body.

3. **Injects frontmatter** — every built file gets: `title`, `index`, `chapter`, `chapterName`, `type` (`recipe` or `technique`), `heroImage`, `referenceImages[]`, `keywords[]`, `cuisine`, `style`, `family`, `course`, `dietary`. Schema defined in `site/src/content.config.ts`.

4. **Copies images** — hero images (`XX-YY.webp/.png`) and reference images (`XX-YYa.webp`, etc.) are copied from chapter folders to `site/public/images/`. Already-optimized WebP files in `site/public/images/` are preserved (not overwritten).

### `heroImage` Frontmatter Format

The `heroImage` field in built files must be the **filename only**: `"07-10.webp"`. Never a path like `"/images/07-10.webp"`.

`[...slug].astro` automatically prepends `/images/` at render time. A full path produces `/images//images/07-10.webp`.

## 5. Image Workflow

### Image Types & Naming

| Type | Filename pattern | Example |
|------|----------------|---------|
| Hero image | `{index}.webp` | `04-16.webp` |
| Reference image | `{index}{letter}.webp` | `04-16a.webp`, `04-16b.webp` |

**Canonical location:** `site/public/images/` — the only location for processed images.

### Hero Image Pipeline

1. **Generate brief** — use `/pipeline [id]` or `Agents/Gemini-Gems/CAFÉ ATHENA - VISUAL DIRECTOR GEM INSTRUCTIONS.md` brief format:
   ```
   Recipe: [dish name]
   Chapter: [chapter name]
   Description: [headnote — key visual elements, texture, color, sauce, garnish]
   Cuisine: [cuisine type]
   Key elements: [3–5 primary visual ingredients or techniques]
   ```
2. **Generate image** — paste brief into Gemini Gem 2 (Visual Director). Outputs PNG.
3. **Process** — Photoshop: remove Gemini watermark, export WebP at 80% quality, 1920×1080.
4. **Place** — move processed `XX-YY.webp` to `site/public/images/`.
5. **Update registry** — set `heroImage: true` and `heroImageOptimized: true` in `The Manual/recipes.json`.
6. **Clean up** — delete working files from `The Manual/` chapter folder once `heroImageOptimized: true`.

## 6. Script Documentation

All scripts run from the repo root. Ollama must be running at `localhost:11434` for AI-powered scripts.

```bash
# Start Ollama
ollama serve

# Pull required models
ollama pull llama3.2      # generation — 2 GB
ollama pull qwen2.5:7b   # detection — 4.7 GB
ollama pull gemma3:4b    # alternative — 3.3 GB
```

### audit.py — Recipe Audit & Repair

Primary local tool. Scans all recipe files against the format standard, identifies structural issues, generates AI fixes for auto-fixable problems, and applies changes only after user approval. Results written to `The Manual/recipes.json` under an `audit` block.

```bash
python3 scripts/audit.py --scan-only              # scan all, no changes
python3 scripts/audit.py --status                 # show audit status rollup
python3 scripts/audit.py --id 04-15              # audit single recipe
python3 scripts/audit.py --chapter 3              # audit entire chapter
python3 scripts/audit.py --pending-only           # recipes not yet audited
python3 scripts/audit.py --deep                   # + LLM-based Mise/Method violation check (qwen2.5:7b)
python3 scripts/audit.py --model gemma3:4b        # override generation model
python3 scripts/audit.py --detect-model qwen2.5:7b  # override detection model
python3 scripts/audit.py --auto-approve           # skip interactive prompts (Claude Code use)
```

**What it detects:**

| Issue | Detection | Auto-fix? |
|-------|-----------|---------|
| Missing `## Headnote` | Structural | No |
| Missing `## Mise en Place` | Structural | No |
| Missing `## Ingredients` | Structural | No |
| Missing `## Method` | Structural | No |
| Missing `## Glossary` | Structural | **Yes** — generates 3–8 culinary definitions |
| Missing `## Keywords` | Structural | **Yes** — generates 10–15 comma-separated terms |
| Missing `## Category` | Structural | **Yes** — assigns from controlled vocabulary |
| Malformed `## Category` | Structural | **Yes** — rewrites to correct format |
| Sections out of order | Structural | No |
| Temperature missing F/C pair | Regex | No |
| Keywords count outside 10–15 | Regex | No |
| Unbolded `Phase N:` in Method | Regex | No |
| Heat step in Mise en Place | LLM (`--deep`) | No |

**`audit` block in `recipes.json`:**

```json
"audit": {
  "lastRun": "2026-06-24",
  "status": "clean | issues_found | approved",
  "issues": ["missing_glossary", "missing_keywords"]
}
```

### markdownlint_safe_fix.py — Deterministic Lint Fixer (Stage 1)

```bash
python3 scripts/markdownlint_safe_fix.py --mode check --verbose --root .
python3 scripts/markdownlint_safe_fix.py --mode dry-run --verbose --root .
python3 scripts/markdownlint_safe_fix.py --mode fix --root .
python3 scripts/markdownlint_safe_fix.py --mode fix --glob "**/04-09.md" --root .
```

No LLM required. Zero content risk — only structural/syntax fixes. Reverts automatically if issue count does not improve.

### fix_markdown_with_ollama.py — Ollama Lint Fixer (Stage 2)

```bash
python3 scripts/fix_markdown_with_ollama.py --dry-run --root .
python3 scripts/fix_markdown_with_ollama.py --root .
python3 scripts/fix_markdown_with_ollama.py --model gemma3:4b --root .
python3 scripts/fix_markdown_with_ollama.py --glob "The Manual/Chapter 4*/**/*.md" --root .
```

For issues the deterministic fixer cannot touch. Requires Ollama. Do not run directly — use the **Markdownlint QA** sub-agent, which handles both stages with authorization checkpoints.

### audit.py vs. /format-audit

| Tool | Scope | Use when |
|------|-------|---------|
| `audit.py` | Structural + regex + optional LLM | Bulk scanning, glossary/keyword generation |
| `audit.py --deep` | + Mise/Method violation detection | Heat-step checking without cloud LLM |
| `/format-audit` | Deep content review via Claude | Sensory cues, complex Mise vs. Method calls, edge cases |

Run `audit.py` first to close structural gaps, then `/format-audit` for nuanced content review.

### Deprecated Scripts

Do not use directly — superseded by `audit.py`:

- `scripts/extract-keywords.py`
- `scripts/add-glossary-sections.py`

## 10. Chef Agent Modes

The Chef operates in three explicit modes. Mode is declared at the start of each session.

### Mode 1 — The Lab (Recipe Development)

- **Goal:** Iterative recipe development with food-science grounding
- **Input:** Concept + rough ingredients
- **Output:** Tested, finalized recipe (not yet formatted)
- **Process:** 1–10 passes of iterative refinement
- **Exit:** User says "Finalize" or "Ready for Manual"

**Starter:**
```
"Let's work on a risotto. I want to focus on starch leaching
and how it creates that creamy texture without cream. Starting
ingredients: Carnaroli rice, homemade stock, white wine, butter, Parmigiano."
```

### Mode 2 — The Manual (Production Formatting)

- **Goal:** Convert a finalized recipe to manuscript-ready format
- **Input:** Finalized recipe from Mode 1
- **Output:** Formatted folio + INDEX DATA block
- **Process:** Single-pass formatting + auto-indexing
- **Requirement:** Chapter must be identified; Claude scans the live Manual directory for the next sequence number

**Starter:**
```
"Format this for publication. Chapter: Chapter 4 - The Mill.
Here's the finalized version: [recipe text]"
```

### Mode 3 — The MasterClass (Technique Education)

- **Goal:** Educate on a culinary technique
- **Input:** Technique topic + specific questions
- **Output:** Tutorial → (on request) Technique Folio
- **Template:** `Guidance/Technique_Folio_Template_v1.md`

**Starter:**
```
"Teach me about the Maillard reaction — not just browning,
the actual chemistry and how temperature affects it."
```

### Critical Constraints (All Modes)

- **Food Safety:** Stop on HACCP violations or unsafe temperatures
- **Format Validation:** Enforce `Guidance/Recipe-Format-Standard.md`
- **Index Integrity:** Scan live Manual directory before assigning any XX-YY number
- **Zero-Citation Protocol:** No `[source]`, `[1]`, `[cite]`, or bracketed references — all content is manuscript-ready

## 11. Indexing System

Every recipe and folio has a unique `XX-YY` index:

- `XX` = Chapter number (zero-padded, `01`–`12`)
- `YY` = Sequential entry number within the chapter

Example: `04-11` = Chapter 4, entry 11.

The hero image filename always matches the folio index: `04-11.webp`. Reference images extend with a letter: `04-11a.webp`, `04-11b.webp`.

**Index assignment:** Always scan the live `The Manual/` directory before assigning a new number. Never infer from `recipes.json` or any cached document — the filesystem is the source of truth.

**New recipe mandatory updates** (all four required before committing):
1. Folio file in `The Manual/Chapter N/XX-YY Café Athena - [Title].md`
2. `The Manual/recipes.json` — register with correct stage flags
3. `The Manual/Cafe-Athena-The-Manual-Current-Version.md` — append under correct chapter heading
4. `PROJECT_STATUS.md` — update Last Updated date

`/register-recipe` handles items 2 and 3 automatically.
